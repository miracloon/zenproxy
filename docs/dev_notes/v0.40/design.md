# v0.40 Design — 管理后台节点工作台化与代理列表减负

> Audience: AI / Dev
> Status: Locked
> 本文记录 v0.40 设计讨论最终共识，作为 plan 阶段输入。

---

## 1. 版本目标

v0.40 聚焦管理后台「订阅与节点管理」页的 UI 与交互重整，目标不是增加新业务能力，而是修复当前已经影响使用效率的工作流问题：

1. 解决代理列表横向过宽、必须频繁左右滚动的问题
2. 解决“操作区”与代理列表脱节、用户无法边看边操作的问题
3. 在不引入前端框架、不重构为卡片列表的前提下，把该页升级为一个更清爽、自然、低心智负担的“节点工作台”
4. 吸收 v0.39 已引入的 `IPv4 / IPv6` 展示与筛选约束，避免新设计在落地时立即过期

---

## 2. 前置影响判断

### 2.1 v0.38 的影响判断

v0.38 当前已实际演变为 remote 客户端启动自动拉取与全量拉取版本，不再是先前讨论中的 UI 版本。

因此，v0.38 **对本次管理后台 UI 设计没有直接结构性约束**，只可作为版本编号变更背景，不构成本次页面设计输入。

### 2.2 v0.39 的影响判断

v0.39 对本次设计有**直接影响**。

原因：

- `ip_family` 已正式进入质量模型
- 管理后台已支持 `IPv4 / IPv6` 质量筛选
- 代理列表已新增 `IP族` 展示列

这意味着当前表格宽度问题相比最初讨论时**进一步加重**，v0.40 的设计不能只做“操作区位置调整”，必须同步重构表格的信息组织方式。

### 2.3 设计结论

v0.40 不采用：

- 纯粹的“操作区搬家”
- 左侧固定操作栏（方案 B）
- 表格改卡片列表（方案 C）

v0.40 采用：

- **A 的强化版：节点工作台化**

即：保留表格心智与现有数据结构，但重组页面结构、工具条语义与列信息密度。

---

## 3. 已确认问题事实

### 3.1 宽度问题的本质

当前代理列表同时展示以下信息维度：

- 名称
- 类型
- 服务器
- 状态
- 端口
- 错误
- IP
- IP族
- 国家
- 风险
- GPT
- Google
- 住宅
- 操作

虽然外层卡片允许横向滚动，但这只是技术性兜底，不是良好的使用体验。用户在查看单行完整信息时必须频繁拖动横向滚动条，阅读与比较成本过高。

### 3.2 操作区问题的本质

当前“操作”作为一个独立大区块放在页面上方，且与“订阅源”“代理列表”之间被切成多个 section。

结果是：

- 用户看完节点信息后，无法就地操作
- 批量操作与当前筛选结果脱节
- 视线与鼠标需要在上方操作区、下方表格之间来回跳跃
- 页面虽然功能很多，但缺少连续的工作流

### 3.3 心智负担问题

当前页面同时存在：

- 顶部四分区全局操作
- 列表多选行为
- 行内四按钮操作
- 类型分布独立 section

这导致用户需要同时理解多套控制面板，而不是围绕“当前这批节点”形成一个统一的工作上下文。

---

## 4. 设计总方案：节点工作台化

### 4.1 核心思路

把当前“代理列表”从一个普通 section，升级为一个完整的**节点工作台（Node Workspace）**。

该工作台内部统一承载：

- 当前结果概览
- 筛选条件
- 当前上下文相关操作
- 列表结果本体

设计目标是让用户在一个连续视线区里完成以下动作：

```text
查看当前结果 → 筛选 / 搜索 → 选中节点 → 立即操作 → 继续观察结果
```

### 4.2 结构原则

- 页面保留原有“统计卡片”“订阅源”能力
- 不再保留独立的大块“操作”section
- “类型分布”不再作为中断工作流的独立 section
- 所有和“当前代理列表”强相关的控制行为都收敛到工作台内部

---

## 5. 页面结构调整

v0.40 中，「订阅与节点管理」页调整为：

```text
统计卡片
订阅源
节点工作台
  ├─ 工作台 Header（标题 / 数量 / 摘要）
  ├─ 筛选层
  ├─ 操作工具条
  └─ 代理表格
```

### 5.1 统计卡片

保留现有统计卡片，不作为本次重点改造对象。

### 5.2 订阅源区

保留现有订阅源新增与列表能力，但视觉权重应略低于节点工作台，避免与代理列表主任务争抢注意力。

### 5.3 类型分布处理

当前“类型分布”独立 section 会打断从“操作”到“代理列表”的连续性。

v0.40 调整为以下二选一策略中的简化方案：

- 优先方案：并入工作台 Header，作为类型 chips 摘要
- 备选方案：保留但做成默认展开较轻的次级卡片，放在工作台标题下方

本版本推荐采用优先方案，不再让其独立占一整段。

---

## 6. 工作台 Header 设计

### 6.1 组成

工作台 Header 应包含：

- 标题：`代理列表`
- 结果总数
- 当前筛选结果数
- 选中数量摘要
- 类型分布 chips（若采用并入方案）

### 6.2 设计目的

Header 不只是标题栏，而是当前工作上下文摘要。

用户一眼应能看到：

- 当前在看多少条代理
- 是否正在筛选
- 是否已有选中项
- 当前列表大致由哪些类型组成

### 6.3 视觉原则

- 比普通 `section-title` 更稳定、更像工作区顶部
- 文字层级清晰，摘要弱于标题
- 不做大面积高对比装饰，保持管理后台克制感

---

## 7. 筛选层设计

### 7.1 保留的核心筛选

v0.40 保留并整合以下筛选项：

- 搜索
- 验证状态
- 启用状态
- 类型
- 质量筛选
- 每页数量

### 7.2 与 v0.39 的兼容要求

质量筛选必须继续保留：

- `ChatGPT 可用`
- `Google 可用`
- `Residential`
- `IPv4 出口`
- `IPv6 出口`

不得因为本次 UI 压缩而回退或弱化 `ip_family` 维度。

### 7.3 视觉定位

筛选层是“条件条”，不是“操作面板”。

设计上应更轻、更扁平，避免和下方操作工具条在视觉上抢同一层级。

---

## 8. 操作工具条设计

### 8.1 设计原则

操作工具条是 v0.40 的核心改造点。

它必须：

- 紧贴代理列表
- 与当前筛选 / 当前选中状态强绑定
- 明确区分“全局动作”和“选中动作”
- 让用户在查看列表时可以直接操作

### 8.2 双状态模型

工具条采用双状态切换，而不是所有按钮永久全展开。

#### 状态 A：未选中模式

当 `selectedIds.size == 0` 时，工具条显示全局动作：

- 验证全部
- 验证未启用
- 验证无效
- 质检全部
- 一键启用有效
- 一键禁用无效
- 清理无效
- 清理三不通

此时工具条左侧显示：

- 当前筛选结果数量
- 当前无选中项的说明

#### 状态 B：选中模式

当 `selectedIds.size > 0` 时，工具条切换为选中动作：

- 启用选中
- 禁用选中
- 验证选中
- 质检选中
- 删除选中

同时显示：

- 已选数量
- 当前筛选结果数
- `取消选择`

### 8.3 为什么必须做状态切换

若继续让全局动作与选中动作同时展开，用户仍然要自己判断“我现在该点哪一组”，心智负担不会真正下降。

双状态切换可以把界面主动收敛到当前任务上下文。

### 8.4 Sticky 策略

桌面端可为该工具条增加轻量 sticky 行为：

- 用户向下滚动长表时，工具条保持在视野上沿附近
- 不是吸顶导航条，不应产生强压迫感
- 移动端不强制 sticky，避免占据有限视口

---

## 9. 表格信息重组

### 9.1 设计原则

v0.40 不靠“继续允许横向滚动”解决宽度问题，而是按用户判断节点时的真实心智，将分散字段重组成更有意义的信息块。

### 9.2 目标列结构

推荐压缩为以下核心列：

1. 选择
2. 名称
3. 节点信息
4. 状态
5. 端口 / 错误
6. 质量标签
7. 操作

### 9.3 列语义定义

#### 名称

- 显示代理名称
- 允许双行或更宽松的文本截断策略
- 不再过度依赖超短单行截断

#### 节点信息

合并以下内容：

- `server:port`
- `IP`
- `IP族`
- `国家`

显示策略采用主副两层：

- 第一层：`server:port`
- 第二层：`IP · IPv4/IPv6 · 国家`

#### 状态

保留 v0.37 已确立的双 badge 语义：

- 有效 / 无效 / 待检测
- 已禁用

必要时允许状态列内部上下排布，而不是强制横向延展。

#### 端口 / 错误

合并以下内容：

- 本地端口
- 错误计数

其中：

- 端口作为主信息
- 错误作为次级信息

#### 质量标签

合并以下内容：

- GPT
- Google
- 住宅
- 风险

设计目标是不再让用户横向逐列扫多个 `Y/N`。

推荐做法：

- GPT / Google / 住宅 使用短标签
- 风险使用单独 badge
- 标签不足时允许自动换行

#### 操作

不再保留当前四按钮常驻横向展开形式。

---

## 10. 行内操作设计

### 10.1 设计目标

行内操作应服务于“单条代理的高频动作”，而不是复制顶部所有能力。

### 10.2 推荐策略

默认只保留 1 到 2 个高频动作：

- `启用 / 禁用`
- `连通测试`

其余动作收进 `更多`：

- 质量检测
- 删除

### 10.3 设计原因

- 当前行内四按钮显著拉宽表格
- 颜色过多会让整页显得吵
- 大部分情况下，用户并不需要每行都直接看到全部动作

### 10.4 危险动作

删除必须继续保留确认机制，但不必永久以红色按钮高频暴露在每一行末尾。

---

## 11. 视觉语言与设计感要求

### 11.1 整体风格

v0.40 不追求“花哨”，而追求：

- 清爽
- 稳定
- 有节奏
- 低噪音

### 11.2 视觉层级

推荐形成三层密度：

1. **Header / 摘要层**：稳定、清晰
2. **筛选层**：轻量、扁平
3. **操作层**：更稳、更有控制感

这三层不能视觉等权，否则用户会感到页面发散。

### 11.3 色彩策略

减少彩色按钮泛滥：

- 主动作保留主色
- 普通操作更多使用中性色按钮
- 危险动作仅在必要时使用红色

### 11.4 标签系统

状态 badge、类型 badge、质量标签应统一：

- 高度
- 圆角
- 留白
- 文本密度

避免现在这种“语义对了，但观感像若干独立零件拼在一起”的状态。

### 11.5 行信息层级

每行信息尽量采用“主信息 + 副信息”模式，而不是把所有字段都当作同一级文本处理。

---

## 12. 未知态与缺省态设计

### 12.1 必须显式处理的情况

由于质量数据并非总是完整存在，v0.40 必须正式设计以下状态，而不是临时用 `-` 兜底了事：

- 未质检
- 仅有部分质量信息
- 有 `ip_address` 但无完整增强信息
- 无 `ip_family`

### 12.2 设计原则

- 未知态应弱于明确肯定或否定信息
- 不能让“未知”视觉上等同于“否”
- `IPv4 / IPv6` 只在已识别时显示，不得凭配置猜测

### 12.3 筛选语义

保留现有 `IPv4 / IPv6` 过滤逻辑，不新增“未识别 IP 族”筛选项。

原因：

- 本版本目标是减负，不是扩展筛选矩阵
- “未知态展示”与“未知态可筛选”属于两个问题

---

## 13. 响应式与落地边界

### 13.1 桌面端

桌面端是 v0.40 的主要优化对象。

应重点保证：

- 不再依赖常态横向滚动
- 工具条与列表保持上下文连续
- 表格信息块层级清晰

### 13.2 移动端

移动端以“不明显变差”为底线，不在 v0.40 追求重新设计。

只要求：

- 筛选层自然换行
- 工具条在窄屏可折行
- 表格不因新结构彻底失控

### 13.3 不做事项

v0.40 明确不做：

- 不引入前端框架
- 不改为卡片列表
- 不实现左侧固定操作栏
- 不新增新的代理业务能力
- 不扩展新的筛选维度
- 不重做用户页整体设计

---

## 14. 风险与防漂移约束

### 14.1 主要风险

- 只改布局，不改信息分组，最终仍然太宽
- 只改操作区位置，不改操作语义，心智负担仍然存在
- 忽略 v0.39 的 `ip_family` 约束，导致设计落地即过期
- 行内操作压缩过度，反而损失常用效率

### 14.2 防漂移约束

执行阶段若出现以下倾向，应视为偏离设计：

- 恢复顶部独立大块“操作区”
- 保留当前四按钮常驻操作列不变
- 保留 `IP / IP族 / 国家 / GPT / Google / 住宅` 全部分列
- 为了节省时间退回“继续靠横向滚动”

---

## 15. 版本边界总结

v0.40 的正式边界如下：

- **必须完成**
  - 把代理列表升级为节点工作台结构
  - 取消独立顶部大块操作区，改为紧贴列表的上下文工具条
  - 实现工具条的“未选中 / 已选中”双状态切换
  - 重组表格列信息，吸收 `ip_family` 带来的额外宽度压力
  - 缩减行内操作列宽度
  - 明确未知质量态的显示策略

- **可以作为实现细节微调**
  - Header 的摘要文案
  - 类型分布 chips 的摆放细节
  - sticky 的具体阈值和样式
  - 行内“更多”菜单的具体呈现形式

- **明确不做**
  - B 方案的左右分栏主布局
  - C 方案的卡片化列表重构
  - 新增业务逻辑、API、状态模型
  - 扩大到用户页的整体工作台化改版

v0.40 本质上是一次**围绕管理后台使用流的 UI 重整**，而不是新功能版本。
