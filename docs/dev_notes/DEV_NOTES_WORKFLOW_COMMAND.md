# DEV_NOTES_WORKFLOW_COMMANDS.md

> Audience: AI / 维护者  
> Status: Normative  
> Canonical source: 本文件是 workflow command 文档的编写与维护规范源。  
> Scope: 本文件只约束 `workflow-design / plan / exec / summary / review` 这组 command 文档的编写、维护与修订；不参与普通开发运行时。

---

## 1. 文档定位与非目标

本文件面向的是 **command 文档层**。  
这一层的文档存在于命令系统中，用来把阶段动作具象化；它们不是项目基础控制矩阵的显式组成部分，也不应成为 AI 理解项目时默认可见的正文层。

本文件负责：

- 规定 workflow command 文档该如何写、该写多厚、该承担什么；
- 规定 command 文档与 `DEV_NOTES_WORKFLOW.md` 的边界；
- 规定五个阶段 command 的通用结构、统一口径与禁止越界事项；
- 确保 command 文档轻、准、单阶段，不反向篡改上位运行时规则。

本文件不负责：

- 代替 `DEV_NOTES_WORKFLOW.md` 规定运行时持续约束；
- 代替 `AGENTS.md`、`INTENT.md`、`WORKFLOW.md`、`SPEC.md`；
- 代替某个版本的 `design.md`、`plan.md`、`summary.md`、`review.md`；
- 记录具体项目、具体版本的事实；
- 把 command 文档层提升为项目主链中的显式正文层。

---

## 2. command 文档的角色

workflow command 文档的作用只有一个：  
**把某个阶段的进入动作、工作方式、输出边界和最小护栏，压缩成一份单次可执行的阶段入口文档。**

它们应该让 AI 在 command 执行时快速进入正确语境，但不应承担整套开发协议。

换句话说：

- `DEV_NOTES_WORKFLOW.md` 定义**正式开发态下长期持续生效的运行时规则**；
- workflow command 文档只定义**这一次进入某个阶段时该怎么开始、做到哪、停在哪里**。

---

## 3. 总体设计原则

1. **一条 command 只对应一个主阶段**  
   一个 command 只服务一个主阶段，不混入后续阶段的正式职责。

2. **只写局部，不重写整套协议**  
   command 文档只补充当前阶段最需要的局部语义，不复制上位运行时正文。

3. **默认读取链路以上位 runtime 为准**  
   command 文档应引用 `@docs/dev_notes/DEV_NOTES_WORKFLOW.md`，并依赖其中定义的默认读取链路；不要在每个 command 中重写一遍完整主链。

4. **不得引入与 runtime 相冲突的新规则**  
   command 文档可以具体化，但不能改写 `DEV_NOTES_WORKFLOW.md` 的阶段边界、职责定义和收口方式。

5. **低噪声、自然语言、轻量排版**  
   command 文档是阶段入口，不是制度汇编；应尽量短、准、自然、可执行。

6. **项目特定 patch 与通用正文分离**  
   若 command 中需要包含项目特有约束，应明确标识为按需 patch，而不是混进通用正文。

7. **输出必须能沉淀到对应文档**  
   一个阶段 command 的输出，应自然沉淀到该阶段对应的版本文档，而不是停留在对话里。

---

## 4. command 文档的推荐结构

推荐保留以下结构，按需裁剪，不强求机械完整：

1. `description` front matter  
2. “你现在正在执行项目级 command：...” 的定位句  
3. 一句阶段语义说明  
4. 规范源说明：引用 `@docs/dev_notes/DEV_NOTES_WORKFLOW.md`  
5. 输入解释与版本识别规则  
6. 阶段核心任务  
7. 沟通 / 交互行为规范  
8. 严格边界  
9. 落文条件与产物规范  
10. 下一步提示（仅提示，不跨阶段执行）

不推荐在 command 文档中出现：

- 完整重写 AGENTS / INTENT / WORKFLOW / SPEC 主链；
- 大段复制 `DEV_NOTES_WORKFLOW.md` 的阶段矩阵与总原则；
- 与当前阶段无关的扩展分支；
- 过重的术语化制度语言；
- 任何“默认继续进入下一阶段”的暗示。

## 5. 五个阶段 command 的专属要求

### 5.1 `workflow-design`

必须强调：

- design 是讨论与收敛态，不是立刻落文态；
- 角色是技术顾问 / 架构讨论者，而不是实施者；
- 优先挖掘核心诉求、边界、价值、风险、取舍；
- 允许给出 2-3 个宏观方案并附带推荐；
- 不主动宣告“已经完全对齐”或“现在可以进入下个阶段”；
- 仅在用户明确要求成文时，才产出 `design.md`。

`design.md` 应主要记录：

- 本次目标；
- 边界与不做清单；
- 风险 / 约束 / 取舍；
- 宏观方案与当前结论。

### 5.2 `workflow-plan`

必须强调：

- plan 是执行基线收敛态，不是代码阶段；
- 若 design 前提明显缺失或不稳定，应先指出缺口，而不是强行锁 plan；
- 需要明确 task 载体选择：单一 `task.md` 还是 `taskNN.md`；
- 需要明确依赖关系、执行顺序、完成标准、关键技术约定；
- `plan.md` 是唯一全局基线，`task*.md` 只承载局部任务边界；
- 仅在用户明确要求锁定时，才正式产出或更新 `plan.md` / `task*.md`。

`plan.md` 应主要记录：

- 版本目标；
- 拆解总览；
- 依赖 / 顺序；
- 全局技术约定；
- 全局验收标准。

`task*.md` 应主要记录：

- 当前任务边界；
- 输入 / 输出；
- 依赖关系；
- 完成标准。

### 5.3 `workflow-exec`

必须强调：

- exec 是纯执行态；
- 没有稳定 `plan.md` 就不开工；
- 必须优先读取 `plan.md` 与当前任务文档；
- 代码实现、测试实现、必要配套文档微调可以做；summary / review 绝不能提前做；
- 不把 `design.md`、`plan.md`、`task*.md` 改成进度流水账；
- 完成一个明确任务单元后，先验证、再 commit、再重读任务文档；
- 完成本次指定范围后按需停顿，不自动冲向下个阶段。

### 5.4 `workflow-summary`

必须强调：

- summary 只做事实归档；
- 输入必须覆盖 `design.md`、`plan.md`、`task*.md`、相关 commit 与验证记录；
- 绝不评价代码质量、绝不代替 review；
- 输出应结构稳定、便于追溯；
- 结束于 `summary.md`，不自动进入 review。

`summary.md` 推荐固定骨架：

1. 版本范围 / 目标摘要  
2. 计划基线概览  
3. 实际完成情况  
4. 偏移与计划外新增  
5. 未完成项  
6. commit / 验证索引  

要求：

- 只写事实与状态，不写“好不好”“应该如何改”；
- 尽量使用“已完成 / 部分完成 / 未完成”等中性描述。

### 5.5 `workflow-review`

必须强调：

- review 是质量审阅态，不是事实归档态，也不是修复态；
- 当用户显式触发 `/workflow-review` 时，应直接进入独立审阅语境；
- 不主动要求用户切换新会话，也不把“是否新会话”当成额外门槛；
- 无论当前是否新会话，都必须保持审阅独立性；
- 绝不顺手修代码，绝不自动发起 patch。

`review.md` 推荐固定骨架：

1. 总体结论  
2. 意图一致性  
3. 功能完成度  
4. 代码与结构质量  
5. 文档一致性  
6. 问题与风险（建议标严重级别 / 优先级）  
7. 建议去向（立即修复 / patch / 下版本）  

要求：

- 可以评价、可以指出问题、可以给去向建议；
- 不重写 summary；
- 不把建议去向写成自动执行动作。

## 6. 固定上下文与加载建议

workflow command 文档建议只显式引用：

- `@docs/dev_notes/DEV_NOTES_WORKFLOW.md`

其余主链文档由 runtime 中定义的默认读取链路补足。  
不要在每个 command 文档中硬编码一遍 `AGENTS.md → INTENT.md → WORKFLOW.md → SPEC.md`，除非该 command 确实需要对人类维护者特别强调某个补读顺序。

当 command 确实需要强调额外文件时，只写当前阶段真正关键的那部分，不重写全部文档矩阵。

---

## 7. 风格与反例

推荐风格：

- 自然语言；
- 结构清晰；
- 短段落、短列表；
- 先给判断，再给说明；
- 让用户读起来像“阶段入口说明”，而不是“制度条款”。

常见反例：

- 把 runtime 整段复制进 command；
- 在 design / plan 阶段抢跑写代码；
- 在 exec 阶段写 summary；
- 在 summary 阶段评价质量；
- 在 review 阶段顺手修复；
- 用项目特定 patch 污染通用 command 正文；
- 写一堆抽象工程黑话，增加理解负担。

---

## 8. 维护检查清单

当你编写或修订某个 workflow command 文档时，至少自检以下问题：

- 它是否只对应一个主阶段？
- 它是否只写了当前阶段真正需要的入口信息？
- 它是否与 `DEV_NOTES_WORKFLOW.md` 保持一致？
- 它是否偷做了后续阶段的职责？
- 它是否过度复制上位文档？
- 它的输出是否能自然沉淀到对应版本文档？
- 它是否在不必要地让 AI 知道太多 command 层自身的存在？
- 它是否把项目特定 patch 与通用规则分开了？

满足这些条件，command 文档层就会稳定、低噪声、可维护。

## 9. 关键澄清

### 为什么 plan 不能被省略

`plan.md` 的价值不是“形式上写一份文档”，而是冻结执行基线。  
没有稳定的 plan，exec 阶段就会边写边改需求，任务边界会模糊，commit 会失去追溯意义，最终 summary 只能被迫替计划兜底。

### 为什么下一个任务必须重新读取任务文档

因为长对话中的上下文记忆容易漂移。  
重新读取 `task.md` / `taskNN.md`，是为了确保：当前任务真的完成、下一个任务边界没有被误读、顺序和依赖没有被跳过。

### 为什么 summary 与 review 必须分开

若二者混在一起：

- summary 会掺入价值判断；
- review 会重复事实归档；
- AI 会在同一语境里既当执行者又当评审者，降低审阅独立性。

因此：`summary.md` 只做事实；`review.md` 才做评价。
