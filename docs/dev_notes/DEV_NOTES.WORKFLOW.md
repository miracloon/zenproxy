# DEV_NOTES_WORKFLOW.md

> Audience: AI  
> Status: Normative  
> Canonical source: 本文件是项目正式开发态下的 AI 运行时协作规范源。  
> Scope: 本文件只定义 `design → plan → exec → summary → review` 五阶段，以及 `docs/dev_notes/<version>/...` 的运行时关系与持续约束。

---

## 1. 文档定位与边界

本文件默认假定：**当前是否进入正式开发态，已经由外层入口文档、用户指令或实际任务状态决定。**  
本文件一旦被加载，就只负责正式开发态下跨轮次持续生效的运行时约束。


## 2. 核心原则

1. **先对齐，再执行**  
   未形成稳定 design / plan 基线前，不进入正式开发。

2. **阶段边界清晰**  
   `design / plan / exec / summary / review` 各自只做自己的事，不混写、不代行。

3. **文档职责单一**  
   不让一份文档同时承担设计说明、执行计划、事实归档和质量审阅。

4. **traceability 优先**  
   开发过程的主要追溯载体是文档边界 + commit 历史，而不是聊天式流水账。

5. **完成一个明确任务单元后先 commit**  
   exec 阶段完成一个明确任务单元后，应先提交，再进入下一个任务单元。

6. **推进以任务文档为准，不以记忆为准**  
   进入下一个任务前，应重新读取 `task.md` / `taskNN.md`，而不是沿着对话惯性继续写。

7. **默认不静默改写已锁定计划**  
   若执行中发生偏移，应显式记录或回到 plan 层重定，而不是悄悄把 `plan.md` 改成事后合理化版本。

8. **summary 与 review 分离**  
   `summary` 只归档事实；`review` 只评估质量；二者不得相互替代。

9. **流程用于控边界，不用于制造负担**  
   本文件的价值是让正式开发态保持阶段清晰、文档清晰、提交清晰，而不是把任何工作都流程化。

---

## 3. 工作流总览

正式开发态下的标准阶段链路为：

```text
design → plan → exec → summary → review
```

这五个阶段分别回答：

- **design**：为什么做、边界在哪里、有哪些关键取舍
- **plan**：按什么基线执行、拆成哪些任务、完成标准是什么
- **exec**：按图施工、验证结果、形成可追溯提交
- **summary**：实际交付了什么、偏移了什么、还有什么未完成
- **review**：做得怎么样、哪里有问题、下一步该如何处理

### 3.1 阶段边界矩阵

| 阶段 | 核心目标 | 允许做什么 | 禁止做什么 | 主要输入 | 主要输出 | 收口条件 |
| --- | --- | --- | --- | --- | --- | --- |
| design | 对齐目标、边界、取舍 | 讨论需求、价值、风险、约束、宏观方案 | 直接进入开发、锁执行计划、做正式总结 | 用户意图、项目控制文档、现状观察 | `design.md` | 方向与边界基本稳定 |
| plan | 形成执行基线 | 拆任务、定依赖、定顺序、定完成标准、锁关键技术约定 | 未锁计划就开发、拿 task 文档代替 plan、把规划写成实现流水 | `design.md`、项目控制文档 | `plan.md`、`task.md` / `taskNN.md` | `plan.md` 已可执行 |
| exec | 按计划开发并提交 | 编码、验证、必要的配套文档微调、commit | 提前产出 `summary.md` / `review.md`、静默改写 plan、跨任务混写 | `plan.md`、当前任务文档 | 代码、commit、必要验证结果 | 本次指定执行范围完成 |
| summary | 事实归档 | 对照计划汇总完成情况、偏移、新增、未完成项，整理 commit / 验证索引 | 评价代码质量、提出改进建议、继续开发 | `plan.md`、`task*.md`、commit、验证结果 | `summary.md` | 事实归档完成 |
| review | 独立审阅质量 | 评估完成度、一致性、代码/结构/文档质量，识别问题与去向 | 重复 summary、顺手修代码、自动发起 patch | `design.md`、`plan.md`、`summary.md`、代码与 commit | `review.md` | 审阅结论完成 |

## 4. 运行时阶段识别与默认读取

### 4.1 阶段识别规则

继续推进前，先判断当前处于哪个阶段。优先级如下：

1. 当前是否正在产出或更新 `review.md`；
2. 当前是否正在产出或更新 `summary.md`；
3. 是否已有已锁定的 `plan.md` 且当前任务在做代码实现；
4. 是否已有未锁定但成型的 `plan.md` / `task*.md`；
5. 否则视为仍处于 design。

若阶段信息冲突，以**最新已确认文档状态**为准，而不是以对话记忆为准。

### 4.2 默认读取链路

进入某阶段时，默认按以下顺序补读：

- **design**：`AGENTS.md → INTENT.md → WORKFLOW.md → SPEC.md → docs/dev_notes/<version>/design.md（若存在）`
- **plan**：`AGENTS.md → INTENT.md → WORKFLOW.md → SPEC.md → design.md → 现有 plan / task 草稿（若存在）`
- **exec**：`AGENTS.md → INTENT.md → WORKFLOW.md → SPEC.md → plan.md → 当前 task 文档`
- **summary**：`plan.md → task.md / taskNN.md → 相关 commit → 必要验证记录`
- **review**：`design.md → plan.md → summary.md → 相关代码 / 文档 / commit`

### 4.3 阶段内自主推进规则

用户参与较少时，AI 可以在当前阶段内自主推进，但必须同时满足：

- 不跨阶段提前完成后续阶段的正式职责；
- 不跳过当前阶段的收口动作；
- 不把模糊记忆当作任务来源；
- 不静默修改已锁定的 `plan.md`；
- 不把大块改动揉成无法追溯的提交；
- 不因为“知道下一步是什么”就自动越过用户尚未显式触发的阶段。

## 5. exec 阶段的推进规则

exec 阶段是最容易漂移的阶段，必须严格遵循以下顺序：

1. 读取 `plan.md` 与当前待执行 `task.md` / `taskNN.md`；
2. 精确圈定本次执行范围；
3. 完成当前明确任务单元；
4. 按项目既定路径完成必要验证；
5. **先 commit**；
6. 再重新读取任务文档，定位下一个待执行任务；
7. 根据用户指令决定继续执行、停顿，或准备进入 summary。

### 5.1 exec 阶段硬约束

- 不把 `design.md`、`plan.md`、`task*.md` 当作进度看板去打勾、写完成状态或补流水账；
- 不在任务边界不清的情况下连续推进多个任务；
- 不提前产出 `summary.md` 或 `review.md`；
- 不只凭记忆衔接下一个任务；
- 不在未说明偏移的情况下扩大范围。

### 5.2 停止自主推进的条件

出现以下情况时，应暂停继续推进，并请求用户裁定，或先把偏移显式化：

- 当前需求发生明显方向变化；
- 原 `plan.md` 的核心边界已不再成立；
- 多个任务文档之间出现冲突；
- 需要跨越原定范围做新功能；
- 执行中发现必须上升到 design / plan 重新讨论的重大取舍；
- 用户明确要求暂停、改线或回到更上游阶段；
- 当前工作实际上只是一次非正式小修，继续沿本工作流推进只会增加负担。

## 6. 版本派生文档体系与职责边界

版本级开发文档统一存放于：

```text
docs/dev_notes/<version>/
```

标准形态如下：

```text
docs/dev_notes/<version>/
  design.md
  plan.md
  task.md            # 简单任务时使用
  task00.md ~ taskNN.md
  summary.md
  review.md
  pitfalls.md        # 按需，可选
```

### 6.1 文档职责总表

| 文档 | 回答什么 | 不回答什么 |
| --- | --- | --- |
| `design.md` | 为什么做、边界在哪里、有哪些关键取舍 | 详细编码步骤、执行流水、质量审阅 |
| `plan.md` | 这次开发按什么基线执行、关键依赖与完成标准是什么 | 事实归档、质量评价、逐条进度流水 |
| `task.md` / `taskNN.md` | 当前任务边界、输入输出、依赖与完成标准 | 全局执行基线、版本结论 |
| `summary.md` | 实际交付了什么、偏移了什么、没完成什么 | 质量评价、改进建议、补开发 |
| `review.md` | 做得怎么样、有什么问题与改进空间、建议去向 | 事实归档、直接修复 |
| `pitfalls.md` | 高复用价值、反直觉、值得沉淀的技术坑点 | 普通执行流水、零碎心得 |

### 6.2 `task.md` 与 `taskNN.md` 的选择

- 任务整体较小、内部耦合较高时，使用单一 `task.md`；
- 任务可自然拆成多个相对独立单元时，使用 `taskNN.md`；
- 无论采用哪种形式，都必须能支持：
  - 明确当前任务；
  - 明确下一个待执行任务；
  - 在 exec 阶段完成一项后先 commit，再切换下一项。

---

## 7. summary / review 的收口规则

- `summary` 只做事实归档，不评价好坏，不顺手提改进方案。
- `review` 只做质量审阅，不重复重写 summary，不顺手修代码。
- `review` 由用户显式触发。  
  AI 不自动从 `summary` 跨入 `review`，也不把是否新会话当作额外门槛或额外任务。
- 若用户在当前会话中显式进入 `review`，AI 仍须保持审阅独立性：
  - 不继续开发；
  - 不自动发起 patch；
  - 不把 review 写成 summary + 修复的混合体。

## 8. 项目特定 patch（按需）

本节只用于补充**该项目独有**、且不适合作为通用规则上提的运行时约束。

适合放在这里的内容包括：

- 项目特定的正式验证路径；
- 项目特定的 commit trailer / 任务编号规则；
- 项目特定的目录、命名或交接要求；
- 项目特定的工具限制、发布限制、回退要求。

不应写入这里的内容包括：

- 通用阶段定义；
- 通用文档职责边界；
- 任何项目都成立的工作流原则；
- 某个版本专属的事实或任务细节。

---

## 9. 最小闭环

对于一个版本的标准五阶段闭环，至少应满足：

1. `design.md` 明确阶段性设计共识；
2. `plan.md` 与 `task*.md` 锁定执行基线；
3. exec 按任务推进，并在**每完成一个明确任务单元后先 commit，再读下一个任务**；
4. `summary.md` 完成事实归档；
5. `review.md` 在用户显式触发后完成独立审阅。

若未完成上述闭环，不应宣称该版本已经完整收口。
