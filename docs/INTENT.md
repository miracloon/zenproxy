## 项目定位

ZenProxy 是一个代理池管理与转发服务，面向需要批量注册、数据采集等并发多 IP 出口的使用场景。

项目由两个模块组成：

- **ZenProxy Server**（Rust + Axum）：服务端。管理代理池的导入、验证、质检、用户认证与请求转发。通过订阅源或手动导入获取代理，自动验证可用性并持续质检（IP 类型、地理位置、风险评分、ChatGPT / Google 可达性），对外提供 REST API 供用户按条件获取代理或转发请求。内置 Web 管理后台，可视化查看代理池状态。
- **sing-box-zenproxy**（Go，修改版 sing-box）：客户端。在官方 sing-box 的 Clash API 基础上新增代理存储、订阅管理、远程 Fetch、批量端口绑定功能。用于将代理池能力搬到用户本地使用——每个代理绑定一个本地端口，下游程序通过不同端口获得不同出口 IP。

**核心价值**：集中管理代理池 + 自动化验证质检 + 动态端口绑定，使下游程序无需关心代理协议细节，只需使用本地 HTTP/SOCKS5 端口即可获得多 IP 并发能力。

---

## 核心目标

1. **多协议代理池管理**：支持 VMess、VLESS、Trojan、Shadowsocks、Hysteria2、SOCKS5、HTTP/HTTPS 协议的统一导入、存储和管理。
2. **多格式订阅导入**：支持 Clash YAML、V2Ray URI、Base64、纯文本等订阅格式的自动检测与解析。
3. **自动化验证与质检**：定时验证代理可用性（标记 Valid / Invalid），通过 ip-api.com、ipinfo.io 获取出口 IP 信息、地理位置、住宅属性、风险评分。
4. **动态端口绑定**：运行时通过 REST API 增删代理绑定，无需重启服务；支持批量操作。
5. **按条件获取代理**：支持按国家、类型、住宅属性、风险评分、ChatGPT / Google 可达性等维度筛选代理。
6. **双部署模式**：
   - 服务端模式：集中管理代理池，提供 Fetch API 和 Relay 转发端点。
   - 客户端模式：从服务端拉取或通过订阅/手动导入代理，在本地创建端口池独立使用。

## 边界与不做清单

- **不是 VPN 客户端**：ZenProxy 管理代理池和端口绑定，不提供系统级全局代理或隧道功能。
- **不是通用网络工具**：专注于代理池编排，不处理 DNS、路由规则或流量审计。
- **不修改 sing-box 核心**：客户端基于 sing-box 修改版，修改集中在 `experimental/clashapi/` 目录下新增的代理管理功能；sing-box 自身的协议实现、路由引擎等核心不动。
- **不跟进上游 sing-box 版本更新**：客户端基于 sing-box 1.13.0 (dev-next 分支)，使用固定版本。

---

## 关键设计决策

1. **客户端基于 sing-box 修改版而非独立实现**：直接复用 sing-box 的多协议支持和动态 inbound/outbound 能力，在 Clash API 上层新增代理管理逻辑。这免去了重新实现协议栈的成本，但也意味着客户端的能力边界受 sing-box 架构约束。
2. **代理存储使用文件而非数据库（客户端）**：客户端使用 `data/store.json` 文件持久化代理列表，简化部署；服务端使用 SQLite（`data/zenproxy.db`）。
3. **端口绑定不持久化（客户端）**：客户端重启后需重新调用 `/bindings/batch` 创建绑定。设计如此是因为 sing-box 的 inbound/outbound 管理本身是内存态。
4. **服务端与 sing-box 同进程部署**：zenproxy server 以子进程方式启动 sing-box，通过 HTTP 调用 Clash API 管理绑定。Docker 环境下，server 镜像内嵌两个二进制文件，不拆分为独立容器。
5. **Docker 容器使用 host 网络模式**：代理池场景下容器需要暴露数百个端口（服务端 300+，客户端 5000+），Docker 的端口映射模式（`-p` / `ports:`）在端口数量大时会导致容器启动和停止严重卡顿。因此容器统一采用 `network_mode: host`，直接使用宿主机网络栈。端口范围由各自配置文件管理（服务端 `config.toml`，客户端环境变量），不通过 Docker 端口映射控制。

## Fork 定位

本仓库 fork 自原始 ZenProxy 项目，保持对上游核心逻辑的跟进，同时进行使用层面的功能增强和体验优化。

**核心原则**：不动核心架构和核心业务逻辑（代理验证、质检、订阅刷新、协议解析等），只在使用层面做增量。

**与上游的关系**：
- 使用 `fetch` 分支完全同步上游。
- 自主开发在 `dev` 分支进行。
- `main` 分支合并 `dev`，作为发布分支（触发 CI 构建镜像）。

## Fork 增量意图

以下是 fork 后计划增加的功能或优化项，均属于使用层增强，不涉及核心架构变更。

### 1. Docker 镜像与 CI/CD

构建两个独立 Docker 镜像，基于 GitHub Actions 自动编译（amd64 + arm64）并推送到 DockerHub：

- **server 镜像**：内嵌 zenproxy + sing-box 二进制（zenproxy 以子进程方式启动 sing-box）。
- **client 镜像**：仅包含 sing-box-zenproxy 二进制。

配置文件和数据通过 volume mount 挂载，容器重建后数据不丢失。

### 2. 客户端端口配置增强

- 允许自定义代理池端口范围（源项目硬编码 20001-30000，fork 默认改为 60001-65535）。
- 客户端支持端口同步模式：Fetch 远端代理时使用远端的绑定端口号创建本地绑定，使同一代理在 VPS 和本地拥有相同端口号。通过环境变量 `SYNC_REMOTE_PORT=true` 全局开启，或 Fetch API 参数按次指定。开启后 `PROXY_PORT_START/END` 可配置但不生效，端口冲突时报错跳过、不回退。
- 服务端重启时优先复用上次的端口分配，保障端口号在代理持续有效期间保持稳定。

### 3. 服务端认证方式扩展

在现有 Linux.do OAuth 认证基础上，新增基本的用户名 / 密码认证方式。作为 OAuth 的补充选项，不替代现有认证体系。

### 4. 配置方案优化

将上述新增配置（端口范围、认证方式等）集成为更优雅、清晰的统一配置文件方案。

### 5. 客户端 Dashboard

在 sing-box-zenproxy 的 Clash API 上嵌入一个单页 HTML Dashboard，访问 API 地址即可在浏览器中可视化查看已拉取的代理节点列表、绑定状态、端口映射等信息。数据源为已有 REST API（`/store`、`/bindings`、`/subscriptions`），不引入额外后端依赖。

### 6. 使用场景与流量链路

#### 部署拓扑

- **VPS**：仅运行 zenproxy server 容器。Server 内嵌的 sing-box 子进程为有效代理创建端口绑定，VPS 上的下游服务直接使用这些端口。
- **本地**：仅运行 sing-box-zenproxy client 容器。从远程 Server Fetch 代理信息后在本地创建端口绑定。

不存在同一台机器同时部署 Server 和 Client 的场景——Server 已经包含了完整的端口绑定能力。

#### 使用模式

- **VPS 直接使用**：VPS 上的下游服务 → `127.0.0.1:(Server 代理池端口)` → Server 内嵌 sing-box 的代理绑定 → 代理服务器 → 目标。
- **本地 Fetch 模式**：本地下游服务 → `127.0.0.1:(Client 代理池端口)` → 本地 sing-box-zenproxy → 代理服务器 → 目标。流量不经过 VPS，仅从 VPS 的 Server 拉取代理池信息。

## 术语与概念澄清

| 术语 | 含义 | 易混淆点 |
| --- | --- | --- |
| 绑定 (Binding) | 一个本地端口与一个代理出站的映射关系 | 不是网络协议层的 bind，是 sing-box 的运行态概念 |
| 代理池端口 | singbox-zenproxy 分配的本地监听端口集合 | 不是代理服务器的远端端口，是本地入口端口 |
| Fetch | 客户端从服务端 API 拉取代理信息（含完整 outbound 配置） | 不是流量转发，只是数据获取 |
| Relay | 服务端 `/api/relay` 端点，代为转发 HTTP 请求 | 与代理池端口无关，是服务端的 HTTP 代理功能 |
| 验证 (Validation) | 检测代理是否可用（能否通过代理访问目标 URL） | 不同于质检 |
| 质检 (Quality Check) | 获取代理出口 IP 的信息：地理位置、IP 类型、风险评分等 | 不同于验证 |
